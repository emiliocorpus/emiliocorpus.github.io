BLOG 8:So You Think You are Classy, Eh?
Sunday, 5/31/2015
I am currently at the very beginning of my programming journey and there is a lot of information I have taken in so far. I have been endowed the ability to create arrays, strings and hashes, as well as the ability to edit them down the road. I thought these concepts were super cool until I recently experienced Classes, which has given me even more power in accessing, creating, or editing certain objects!

I think the best way to really explain Classes is by example. I am a 20-odd year old male who has so many favorite things in this world that I cannot keep track. Let's create a class that can build a collection of my favorite things!

class Favorites
     def initialize
          @favorite_items = {}
     end


     def favorite_items
          @favorite_items
     end


     def add_fav (category,item)
          @favorite_items[category] = item
     end
end


I have just created a class, albeit, a simple one! Can you guess it's functionality? Lets take it step by step, unwrapping the code from the outside going inwards!

class Favorites


          # Some code in here


end


The outer part of this code is actually creating a class called Favorites. If we were to call declare a new variable emilio = Favorites.new and called emilio.class the console would tell us that this object has a class of Favorites. Pretty simple, right? Lets go 1 level further, from top to bottom.

class Favorites
     def initialize
          @favorite_items = {}
     end


end


Defining a method called initialize will make it so if you every time you define a new Favorites object, that object will run the code defined in the initialized code block. For this class, the class sets an instance variable equal to an empty hash. You will see in a few minutes what an instance variable is. But moving on..

class Favorites
     def initialize
          @favorite_items = {}
     end


     def favorite_items
          @favorite_items
     end

end


You are probably asking yourself, what the hell that @ doing in front of favorite_items in the code block. Well, when creating classes, putting an @ in front of a variable lets Ruby know that that variable is an instance variable. The instance variable enables individual objects to remember state. Another cool thing about it is if an instance variable is defined in one method inside a class, it can then be used by any method defined within that class. So when I call emilio.favorite_items, it would return those favorite items. It won't return anything, however, until we go over the next method. You are also probably wondering why the @favorite_items is in the initialize method as well as the favorite_items. If you wanted to return the collection, i.e., favorite_items, you must define a method that returns it, and thats what we did with the second method created in this class!**

**although there is a shortcut to this, which I will fill you in about down the road!

class Favorites
     def initialize
          @favorite_items = {}
     end


     def favorite_items
          @favorite_items
     end


     def add_fav (category,item)
          @favorite_items[category] = item
     end
end


Finally, the last method call in this Class add_fav, accepts two parameters, a category and an item. This method functions to create/overwrite a key-value pair, where the category represents the key, while the item represents the pair! So if I wanted to jot down several favorite things using this class, I would do the following code...

emilio = Favorites.new
emilio.add_fav(:movie,"The Italian Job")
p emilio.favorite_items
     # {:movie=>"The Italian Job"}
emilio.add_fav(:movie,"Mad Maxx:Fury Road")
p emilio.favorite_items
     # {:movie=>"Mad Maxx:Fury Road"}
      # This overwrites my previous movie!
emilio.add_fav(:book,"Well-Grounded Rubyist")
p emilio.favorite_items
     # {:movie=>"Mad Maxx:Fury Road", :book=>"Well-Grounded Rubyist"}
p emilio.favorite_items[:book]
     # "Well-Grounded Rubyist"
This is how you would access a certain value given a correct key
p emilio.favorite_items[:sport]
     # This will return nil since that key-value does not exist, yet!


I hope this was clearly conveyed. Although the code is short, there is still ways to refactor and consolidate it. Classes have the ability to declare attributes using the attr_ syntax. There are 3 different types of attributes: attr_reader, attr_writer and attr_accessor. attr_reader attributes are similar to writing the following code...

def some_method
     @value
end


This reader attribute is similar to our favorite_items method! You cannot edit, but when called it will return a value! attr_writer is written like so...

def some_method=(value)
     @value
end


This means you are able to edit/change the instance variable (in this case @value), but this method will not be able to return that method to you if you were to call Class#some_method. The only attribute that can both read and write is attr_accessor, which in code is a combination of both reader/writer would look like this...

def some_method
     @value
end

def some_method=(value)
     @value
end


So in terms of consolidating/refactoring our current class(Favorites), you would use the attr_ like so... Before you looking down, try and guess to see which attr_ we would use.




class Favorites
     attr_reader :favorite_items

     def initialize
          @favorite_items = {}
     end


     def add_fav (category,item)
          @favorite_items[category] = item
     end
end
